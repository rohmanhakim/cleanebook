# Learning Doc: Editor Route (Task 1I)

## Overview

Implemented the editor route that displays job metadata after a user uploads a PDF. This establishes the complete upload → redirect → display flow and proves the foundation for future PDF viewer and region editor features.

## Key Learnings

### 1. SvelteKit Route Groups and Layout Server

The `(app)` route group uses a layout server to provide common behavior:

```typescript
// src/routes/(app)/+layout.server.ts
export const load: LayoutServerLoad = async ({ locals }) => {
  // Pass user to page - anonymous users are valid
  // No auth redirect needed for this route group
  return {
    user: locals.user,
  };
};
```

Key points:
- **No auth redirect** - Anonymous users are valid for the (app) group
- **Pass-through pattern** - Just returns user data from `locals`
- Different from (auth) routes which would redirect to login

### 2. Page Server Load Function with Ownership Validation

The editor page server validates job ownership:

```typescript
// src/routes/(app)/editor/[jobId]/+page.server.ts
export const load: PageServerLoad = async ({ params, locals, platform }) => {
  const { jobId } = params;
  const user = locals.user;

  // User should exist here due to hooks.server.ts
  if (!user) {
    error(404, { message: 'Job not found' });
  }

  const db = platform?.env?.DB;
  if (!db) {
    error(500, { message: 'Database not available' });
  }

  const job = await getJobById(db, jobId);

  // Job not found or user doesn't own it
  // Return 404 for both cases (don't reveal if job exists)
  if (!job || job.userId !== user.id) {
    error(404, { message: 'Job not found' });
  }

  return { job };
};
```

Security considerations:
- **404 for both cases** - Don't reveal if a job exists to unauthorized users
- **Ownership check** - `job.userId === user.id` ensures users can only see their own jobs
- **Graceful handling** - Handle null user and missing database

### 3. Detached ArrayBuffer Issue

When uploading PDFs, encountered `TypeError: Cannot perform Construct on a detached ArrayBuffer`:

```
Failed to store file in R2: TypeError: Cannot perform Construct on a detached ArrayBuffer
    at new Uint8Array (<anonymous>)
    ...
```

**Root Cause**: `pdfjs-serverless` detaches the ArrayBuffer during page count extraction, making it unusable for subsequent R2 storage.

**Solution**: Re-read the file for R2 storage:

```typescript
// Before (broken)
await platform!.env.R2.put(r2Key, arrayBuffer); // arrayBuffer is detached!

// After (fixed)
const r2Buffer = await file.arrayBuffer(); // Fresh read
await platform!.env.R2.put(r2Key, r2Buffer);
```

### 4. Svelte 5 Component with Props

The page component uses Svelte 5's `$props()` rune:

```svelte
<script lang="ts">
  import type { PageData } from './$types';

  let { data }: { data: PageData } = $props();

  // Access job data
  const job = data.job;
</script>
```

Key points:
- `$props()` replaces the older `export let data` syntax
- Type annotation ensures type safety
- `PageData` type is generated by SvelteKit

### 5. Anonymous User Flow

The complete flow for anonymous users:

1. **Landing page** - User visits `/`, no session yet
2. **Upload PDF** - POST to `/api/upload` triggers anonymous user creation in `hooks.server.ts`
3. **Session created** - Anonymous user + session cookie set
4. **Job created** - Job record links to anonymous user
5. **Redirect to editor** - User redirected to `/editor/{jobId}`
6. **Page loads** - Job data displayed with user's session

### 6. Direct Navigation Scenarios

| User Navigates To | Result |
|-------------------|--------|
| `/editor` (no jobId) | 404 - No route matches |
| `/editor/invalid-id` | 404 - Job not found in D1 |
| `/editor/valid-job-id` (not owned) | 404 - Ownership check fails |
| `/editor/valid-job-id` (owned) | 200 - Job data displayed |

Security: All failure cases return 404, never revealing if a job exists.

## Pitfalls to Avoid

### 1. Forgetting to Handle Null User

Always check `locals.user` before using it:

```typescript
// Bad - will crash if user is null
const userId = locals.user.id;

// Good - handle null case
if (!locals.user) {
  error(404, { message: 'Job not found' });
}
const userId = locals.user.id;
```

### 2. Revealing Job Existence

Don't use different error codes for "job not found" vs "not authorized":

```typescript
// Bad - reveals job exists
if (!job) {
  error(404, { message: 'Job not found' });
}
if (job.userId !== user.id) {
  error(403, { message: 'Not authorized' }); // 403 reveals existence!
}

// Good - same error for both
if (!job || job.userId !== user.id) {
  error(404, { message: 'Job not found' });
}
```

### 3. ArrayBuffer Reuse After Processing

Some libraries (like pdfjs-serverless) detach ArrayBuffers:

```typescript
// Bad - arrayBuffer may be detached after getPdfPageCount
const arrayBuffer = await file.arrayBuffer();
await getPdfPageCount(arrayBuffer);
await r2.put(key, arrayBuffer); // May fail!

// Good - re-read for each use
const arrayBuffer1 = await file.arrayBuffer();
await getPdfPageCount(arrayBuffer1);
const arrayBuffer2 = await file.arrayBuffer();
await r2.put(key, arrayBuffer2);
```
